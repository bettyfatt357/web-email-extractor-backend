PROPOSED ARCHITECTURE:

Route Structure:
  /                          → Landing page (new)
  /auth/login               → Login form (new)
  /auth/register            → Register form (new)
  /auth/forgot-password     → Forgot password form (new)
  /auth/reset-password      → Reset password form (new)
  /dashboard/*              → Protected dashboard (requires session)
  /admin/*                  → Protected admin (requires session + admin role)

Authentication Flow:
  1. User registers with email/password
  2. Server stores hashed password in database
  3. User logs in with email/password
  4. Server validates and creates session token
  5. Session stored as httpOnly cookie (secure, can't be stolen via JS)
  6. Each request automatically sends cookie
  7. Server validates session on protected routes
  8. If invalid → redirect to login
  9. User can logout (deletes session)

RECOMMENDED DECISIONS:

1. Database:
   - Recommended: Supabase (has auth built-in) or Neon (PostgreSQL)
   - Action: Set up database with provided schema

2. Email Service:
   - Recommended: Resend (modern, Next.js friendly)
   - Alternative: SendGrid, Mailgun
   - Action: Set up email service account

3. Session Duration:
   - Recommended: 24 hours
   - Option: Remember me for 30 days

4. Password Requirements:
   - Recommended: Min 8 chars, must have upper+lower+number+symbol

5. Rate Limiting:
   - Login: 5 failures in 15 mins
   - Register: 3 per IP in 1 hour
   - Password reset: 5 per email in 1 hour

SECURITY MEASURES:

✓ Passwords hashed with bcrypt (not stored plaintext)
✓ httpOnly cookies (can't be stolen via JavaScript)
✓ Secure flag (HTTPS only)
✓ SameSite=Strict (CSRF protection)
✓ Session expiry (24 hours default)
✓ Password reset tokens expire (1 hour)
✓ Rate limiting on auth endpoints
✓ Constant-time password comparison (no timing attacks)
